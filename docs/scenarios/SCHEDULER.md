# SCHEDULER Scenario

## 목적

이 문서는 스케줄러가 시간이 흐름에 따라 센서의 상태를 모니터링하고, 일정 시간 데이터가 없는 센서를 FAULTY 상태로 전환하는 과정을 설명한다.
스케줄러는 데이터 수집 요청과는 별개로 주기적으로 실행되어 "센서가 살아 있는지"를 지속적으로 확인하는 역할을 한다.
서술 안에서는 Phase 6 구현에서 사용되는 함수와 설정 이름을 그대로 언급해 실제 코드와의 연결점을 분명히 한다.

## 시나리오 1. 시간이 흐른 후 NORMAL 모드 센서가 FAULTY로 전환된다

2024년 3월 15일 오전 8시 55분, `SN-NORMAL-01` 센서가 마지막으로 데이터를 보냈다.
이 센서는 `NORMAL` 모드로 동작하고 있었고, 당시 상태는 `HEALTHY`였다.

그런데 문제가 발생했다.
현장에서 네트워크 장애가 생겨 센서가 서버와 통신할 수 없게 되었다.
센서 자체는 계속 측정하고 있지만, 서버로 전송되지 않고 있다.

13분이 흐른 오전 9시 8분, 스케줄러가 실행 주기(10초)에 따라 `HealthEvaluationJob.evaluate_all_sensors()`를 호출한다.
이 함수는 등록된 모든 센서를 순회하며 각 센서의 마지막 데이터 수신 시각과 현재 시각을 비교한다.

`SN-NORMAL-01`에 대해 스케줄러는 다음 계산을 수행한다.
현재 시각(9:08:00) - 마지막 수신 시각(8:55:00) = 13분.
`StatusThresholds.normal_health` 값은 12분(720초)이므로, 13분은 임계값을 초과했다.

스케줄러는 이 센서의 상태를 `HEALTHY`에서 `FAULTY`로 변경한다.
이때 구조화된 로그 `sensor_health_transition`이 기록되며, 어떤 센서가 언제 어떤 이유로 상태가 바뀌었는지 추적할 수 있다.

운영자가 대시보드를 확인하면 `SN-NORMAL-01`이 빨간색으로 표시되며, 13분 전부터 데이터가 없었다는 사실을 알 수 있다.
문제의 원인을 파악해 현장에 나가 네트워크를 복구하기로 결정한다.

## 시나리오 2. EMERGENCY 모드 센서는 더 빠르게 FAULTY로 전환된다

오전 9시 15분, 비상 상황이 발생했다.
`SN-EMERGENCY-01` 센서는 `EMERGENCY` 모드로 동작 중이었는데, 이 센서도 통신이 두절되었다.

`EMERGENCY` 모드는 비상 상황을 대비해 더 민감하게 모니터링되어야 한다.
`StatusThresholds.emergency_health` 값은 30초로 설정되어 있어, `NORMAL` 모드의 12분과는 비교할 수 없을 정도로 짧다.

35초가 지난 오전 9시 15분 35초, 스케줄러가 다시 실행된다.
`HealthEvaluationJob.evaluate_all_sensors()`는 `SN-EMERGENCY-01`의 경과 시간을 계산하고, 30초 임계값을 초과했음을 확인한다.

센서는 즉시 `FAULTY` 상태로 전환된다.
`NORMAL` 모드 센서라면 12분 동안 기다렸겠지만, `EMERGENCY` 모드는 30초 만에 문제를 감지했다.
운영자는 대시보드에서 빨간색 알림을 보고 즉시 대응팀을 출동시킨다.

이 시나리오는 센서 모드에 따라 다른 시간 민감도를 적용하는 것이 왜 중요한지 보여준다.
비상 상황에서는 빠른 감지가 생명과 직결될 수 있다.

## 시나리오 3. 새로운 데이터 수신으로 FAULTY에서 복구된다

오전 9시 20분, 현장 기술자가 네트워크 장비를 교체했다.
`SN-NORMAL-01` 센서가 다시 서버와 통신할 수 있게 된 것이다.

센서는 즉시 측정 데이터를 전송하고, 서버는 `handle_ingest_request(payload, ingest_mode)`로 이를 받는다.
데이터 저장 후 `update_current_sensor_status(records)`가 실행되면서 센서의 `last_received_at`이 현재 시각(9:20:00)으로 갱신된다.

여기서 중요한 점은 상태가 아직 `FAULTY`인 채로 저장된다는 것이다.
스케줄러가 아닌 수집 API는 상태를 HEALTHY로 바로 바꾸지 않는다.
대신 마지막 수신 시각만 업데이트하고, 다음 스케줄러 실행 시점에 건강 상태를 다시 평가하도록 한다.

10초 후 오전 9시 20분 10초, 스케줄러가 실행된다.
`HealthEvaluationJob.evaluate_all_sensors()`는 `SN-NORMAL-01`의 경과 시간을 계산한다.
현재 시각(9:20:10) - 마지막 수신 시각(9:20:00) = 10초.
10초는 `StatusThresholds.normal_health`(12분)보다 훨씬 작으므로, 센서는 `HEALTHY`로 상태가 전환된다.

`sensor_health_transition` 로그가 다시 기록되며, 이번에는 `FAULTY`에서 `HEALTHY`로 복구되었음을 알린다.
운영자의 대시보드에서 빨간색 표시가 녹색으로 바뀌고, "센서가 정상적으로 복구되었습니다"라는 알림이 표시된다.

이 시나리오는 스케줄러가 상태 전환의 유일한 결정자임을 보여준다.
데이터 수집은 사실(마지막 수신 시각)만 기록하고, 상태 판단은 스케줄러가 주기적으로 수행한다.

## 시나리오 4. 스케줄러가 일부 센서에서 예외를 만나도 계속 실행된다

오전 9시 30분, 스케줄러가 실행되었다.
현재 시스템에는 100개의 센서가 등록되어 있다.

`HealthEvaluationJob.evaluate_all_sensors()`는 센서 목록을 순회하기 시작한다.
1번째 센서, 2번째 센서... 순조롭게 진행된다.
그런데 50번째 센서(`SN-UNKNOWN-50`)를 평가하던 중 예외가 발생했다.
이 센서의 레코드에 예상치 못한 NULL 값이 있어 상태 계산 로직에서 오류가 난 것이다.

여기서 중요한 것은 스케줄러의 복원력(resilience)이다.
`evaluate_all_sensors()` 함수는 각 센서를 평가할 때 개별 try-catch 블록으로 감싸고 있다.
50번째 센서에서 예외가 발생해도 이를 catch하여 `health_evaluation_job_failed` 로그로 기록하고, 다음 센서(51번째)로 계속 진행한다.

결과적으로 100개 센서 중 99개는 정상 평가되고, 1개(50번째)는 실패로 기록된다.
함수는 `total_sensors`, `evaluated_count`, `failed_count`가 포함된 결과 객체를 반환한다.

이 방식의 장점은 한 센서의 문제가 전체 스케줄러를 중단시키지 않는다는 것이다.
만약 예외를 catch하지 않았다면, 50번째 센서에서 스케줄러가 멈추고 51~100번째 센서는 평가되지 않은 채로 남게 된다.

운영자는 로그에서 `SN-UNKNOWN-50`의 평가 실패를 확인하고, 해당 센서의 데이터를 점검하기로 한다.
하지만 다른 99개 센서는 정상적으로 모니터링되고 있으므로 전체 시스템 가용성에는 영향이 없다.

## 시나리오 5. Graceful Shutdown으로 실행 중인 job이 안전하게 완료된다

오전 10시, 새로운 기능 배포를 위해 서버를 재시작해야 한다.
운영자는 무중단 배포를 위해 현재 실행 중인 작업들이 완료될 때까지 기다리기로 결정했다.

FastAPI의 lifespan 이벤트 시스템이 작동한다.
애플리케이션 종료 시 `shutdown` 이벤트가 발생하고, 이 이벤트 핸들러는 `SchedulerRunner.stop()`을 호출한다.

`SchedulerRunner`는 APScheduler 기반으로, `SCHEDULER_HEALTH_EVALUATION_INTERVAL_SECONDS` 설정(10초)에 따라 주기적으로 job을 실행한다.
현재 9시 59분 55초에 시작된 evaluation job이 아직 실행 중이다.
이 job은 100개 센서 중 75번째까지 평가했고, 76번째부터 계속 진행 중이다.

`SchedulerRunner.stop()`은 APScheduler의 `shutdown(wait=True)`을 호출한다.
이는 현재 실행 중인 job이 완료될 때까지 최대 30초간 기다린다는 의미이다.

스케줄러는 76~100번째 센서를 계속 평가한다.
오전 10시 00분 05초, 마지막 센서까지 평가가 완료되었다.
이제 스케줄러는 안전하게 종료될 수 있다.

`max_instances=1` 설정도 중요한 역할을 한다.
이 설정은 동일한 job이 동시에 여러 인스턴스로 실행되는 것을 방지한다.
만약 이 설정이 없었다면, 종료 직전에 새로운 job이 시작되어 데이터 일관성 문제가 발생할 수 있었다.

새 서버가 기동되면 FastAPI lifespan의 `startup` 이벤트가 발생하고, `SchedulerRunner`가 다시 초기화된다.
`SCHEDULER_ENABLED` 설정이 `True`이므로 스케줄러는 바로 실행을 시작한다.

운영자는 로그에서 "Scheduler stopped gracefully" 메시지를 확인하고, 새 서버가 정상적으로 기동되었음을 확신한다.
중간에 중단되었던 job은 없었고, 모든 센서가 한 주기 전부 평가되었다.

## 정리

스케줄러는 데이터 수집과는 별개의 시간 축에서 동작한다.
`handle_ingest_request`가 센서로부터 데이터를 받아 저장하는 동안, `HealthEvaluationJob.evaluate_all_sensors()`는 주기적으로 "마지막 데이터가 언제였는지"를 확인한다.

이 두 축의 교차점이 `last_received_at` 필드이다.
수집 API는 이 필드를 업데이트하고, 스케줄러는 이 필드를 읽어 `StatusThresholds`의 임계값과 비교한다.
`NORMAL` 모드는 12분, `EMERGENCY` 모드는 30초라는 다른 기준이 적용되며, 초과 시 `sensor_health_transition` 로그와 함께 상태가 전환된다.

스케줄러의 견고함은 `evaluate_all_sensors()`의 per-sensor error handling에서 나온다.
일부 센서에서 예외가 발생해도 `health_evaluation_job_failed` 로그로 기록하고 계속 진행하여, 전체 모니터링 시스템의 가용성을 유지한다.

그리고 `SchedulerRunner`의 graceful shutdown은 운영 안정성을 보장한다.
FastAPI lifespan 이벤트와 연계되어 서버 재시작 시에도 실행 중인 job이 중단되지 않고 완료될 수 있도록 한다.

즉, 이 스케줄러는 센서가 정상적으로 데이터를 보내는 상황에서는 보이지 않지만,
문제가 발생했을 때 운영자가 인지할 수 있게 만드는 "운영적 보정(operational correction)" 메커니즘이다.
