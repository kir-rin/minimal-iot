# 1. Specification

## Content

### 프로젝트 개요

이 프로젝트는 특정 구역에 배치된 IoT 센서로부터 환경 데이터를 수집하고, 서로 다른 타임존으로 전달되는 시각 정보를 정규화하며, 수집된 데이터를 안정적으로 저장하고, 조건 기반 조회 API를 제공하고, 센서의 상태를 주기 기반으로 판단하며, 외부 제어에 의한 모드 변경 요청을 처리할 수 있는 백엔드 중심 시스템을 정의한다.

이 시스템은 기본적인 웹 서비스 환경을 전제로 하며, 메시지 큐 기반 아키텍처는 사용하지 않는다. 이 과제의 핵심은 단순한 처리량 자체보다도, 단건/배치 혼합 입력, UTC/KST 혼재 timestamp, 센서 모드에 따라 달라지는 운영 규칙을 안정적으로 다루는 데 있다.

### 프로젝트 목표

시스템의 목표는 다음과 같다.

- HTTP REST 기반으로 단건 및 배치 센서 데이터를 수집한다.
- 센서가 UTC 또는 KST로 전송한 `timestamp`를 서버 내부의 일관된 기준 시각으로 정규화한다.
- 측정 데이터와 운영상 필요한 메타데이터를 함께 저장한다.
- 시리얼 번호, 모드, 시간 범위 기준의 조회 API를 제공한다.
- 데이터 누락 여부를 기준으로 센서의 정상/이상 상태를 판별한다.
- 서버가 센서의 작동 모드 변경을 요청할 수 있도록 제어 API를 제공한다.

### 대상 사용자

이 백엔드 시스템의 주요 사용자는 다음과 같다.

- 센서 데이터와 상태를 시각화하는 프론트엔드 애플리케이션
- 최근 측정값 조회, 고장 센서 식별, 모드 변경 요청을 수행하는 운영자
- 센서 또는 게이트웨이를 시뮬레이션하여 데이터를 전송하는 테스트 클라이언트
- 수집, 시간 처리, 상태 판별 로직의 설계 품질을 검토하는 과제 평가자

### 목표와 범위

#### 범위 포함

- 단일 객체 및 배열 payload를 모두 지원하는 텔레메트리 수집 API
- 필수 필드 존재 여부 및 타입 검증
- 혼재된 타임존 timestamp의 단일 기준 시각 정규화
- 정규화된 센서 데이터의 영속 저장
- 시리얼 번호, 모드, 센서 생성 시각, 서버 수신 시각 기준의 조회 API
- 모드별 기대 주기에 따른 센서 상태 판별
- 서버 측 모드 변경 요청 기록 API
- API 계약, 저장 구조, 운영 규칙에 대한 문서화

#### 범위 제외

- MQTT 또는 메시지 큐 기반 수집 구조
- 실시간 스트리밍 처리 플랫폼
- 기본적인 API 수준을 넘어서는 복잡한 디바이스 인증 프로토콜
- 데이터 누락 감지를 넘어서는 고급 이상 탐지 엔진
- 전체 프로덕션 배포 자동화

### 기능 요구사항

#### FR-1. 센서 데이터 수신

시스템은 다음 두 가지 형태의 입력을 모두 받을 수 있는 수집 API를 제공해야 한다.

- 단일 센서 측정 객체
- 여러 센서 측정 객체로 구성된 배열

각 레코드는 다음 필드를 포함해야 한다.

- `serial_number`
- `timestamp`
- `mode`
- `temperature`
- `humidity`
- `pressure`
- `location.lat`
- `location.lng`
- `air_quality`

#### FR-2. Payload 구조 및 값 검증

시스템은 다음 항목을 검증해야 한다.

- 최상위 payload 형태가 객체 또는 배열인지 여부
- 필수 필드 존재 여부
- 각 필드의 타입 적합성
- `mode`가 `NORMAL`, `EMERGENCY` 중 하나인지 여부
- `timestamp`가 timezone 정보를 포함한 유효한 ISO8601 형식인지 여부
- 위치 정보와 좌표값이 존재하며 수치적으로 유효한지 여부

검증 오류는 가능한 경우 어떤 레코드에서 문제가 발생했는지 식별할 수 있는 구조화된 응답으로 반환해야 한다.

#### FR-3. 배치 수집 정책 선택 지원

배치 수집 API는 요청 단위로 수집 정책을 선택할 수 있어야 한다.

- 기본 정책: `atomic`
- 선택 정책: `partial`

정책별 동작은 다음과 같다.

- `atomic`: 배치 내 하나의 레코드라도 검증 실패 또는 저장 실패가 발생하면 전체 배치를 실패로 처리하고 아무 데이터도 저장하지 않는다.
- `partial`: 유효한 레코드는 저장하고, 유효하지 않은 레코드는 개별적으로 거절하며, 응답에는 성공 건수, 실패 건수, 레코드별 오류 정보가 포함된다.

단, malformed JSON, 지원하지 않는 최상위 payload 타입 등 요청 자체가 성립하지 않는 경우에는 선택한 정책과 관계없이 전체 요청을 실패로 처리한다.

#### FR-4. 혼재된 타임존 timestamp 정규화

시스템은 일부 센서 데이터가 UTC(`Z`)로, 일부 데이터가 KST(`+09:00`)로 전달되더라도 이를 모두 파싱할 수 있어야 한다.

서비스는 입력된 시각을 내부의 단일 기준 표현으로 변환하여 저장하고 비교해야 하며, 이후 필터링과 비즈니스 로직은 정규화된 시각을 기준으로 수행해야 한다.

#### FR-5. 운영상 필요한 시간 필드 보존

성공적으로 저장된 각 레코드에 대해 시스템은 다음 정보를 함께 보존해야 한다.

- 정규화된 센서 생성 시각
- 서버 수신 시각
- 센서가 보낸 원본 `timestamp` 문자열

이 분리가 필요한 이유는 다음과 같다.

- 센서 생성 시각은 실제 측정 시점의 진실값이다.
- 서버 수신 시각은 수집 지연과 서버 관점 필터링에 필요하다.
- 원본 문자열은 mixed timezone 문제를 추적하고 디버깅하는 데 유용하다.

#### FR-6. 센서 측정값 저장

시스템은 승인된 각 레코드에 대해 측정값과 위치 정보를 저장해야 한다. 저장된 필드는 원본 payload를 재구성할 수 있고, 최근 센서 동작을 판단할 수 있으며, 조회 API 응답에 활용할 수 있을 정도로 충분해야 한다.

#### FR-9. 센서 모드 제어 지원

시스템은 특정 센서를 대상으로 모드 변경 요청을 기록할 수 있는 API를 제공해야 한다.

공통 시나리오에 따르면 센서는 서버 명령에 의해 모드가 변경될 수 있으므로, 백엔드는 다음 두 개념을 분리해야 한다.

- 서버가 요청한 최신 모드
- 센서 텔레메트리가 실제로 보고한 최신 모드

이를 통해 서버 요청이 즉시 센서에 반영되었다고 가정하는 오류를 피할 수 있다.

#### FR-10. 활용 가능한 오류 정보 반환

시스템은 요청 실패 및 레코드 단위 실패에 대해 기계적으로 해석 가능한 오류 정보를 반환해야 한다. 응답은 운영자나 클라이언트가 실패 원인을 이해하고 재시도가 안전한지 판단할 수 있도록 도와야 한다.

### 비기능 요구사항

#### NFR-1. 데이터 무결성

승인된 레코드는 문서화된 정규화 규칙 외에는 조용히 변형되지 않아야 하며, 필수 필드가 암묵적으로 누락되어서는 안 된다.

#### NFR-2. 시간 일관성

내부 비교, 정렬, 필터링, 상태 판별은 모두 하나의 정규화된 시간 기준으로 동작해야 한다. mixed timezone 입력으로 인해 잘못된 순서나 오탐지가 발생해서는 안 된다.

#### NFR-3. 낮은 운영 복잡도

시스템은 메시지 큐나 분산 이벤트 백본 없이도 과제 요구사항을 만족해야 한다. 설계는 기본적인 웹 서비스 환경에서 이해 가능하고 실행 가능해야 한다.

#### NFR-4. 추적 가능성

백엔드는 raw timestamp, 검증 오류, 서버 수신 시각 등 수집 판단을 추적할 수 있는 충분한 정보를 보존해야 한다.

#### NFR-5. 확장 가능성

설계는 추가 모드, 새로운 측정 지표, 더 강한 idempotency 제어, 고도화된 알림 기능을 나중에 확장할 수 있어야 한다.

#### NFR-6. 예측 가능한 API 계약

배치 처리 의미, 필터 의미, 상태 판별 기준은 구현에 암묵적으로 숨기지 않고 문서에 명시되어야 한다.

#### NFR-7. 컨테이너 친화적 실행

제출 가이드에서 Docker 기반 실행을 권장하므로, 프로젝트는 컨테이너 환경에서 실행 가능하도록 구성되는 것이 바람직하다.

### 사용자 시나리오 및 플로우

#### 시나리오 1. 단건 데이터 수집

1. `NORMAL` 모드의 센서가 하나의 JSON 객체를 전송한다.
2. 백엔드는 payload를 검증한다.
3. 백엔드는 mixed timezone timestamp를 파싱한다.
4. 백엔드는 정규화된 시각과 서버 수신 시각을 함께 저장한다.
5. 백엔드는 센서의 최신 상태 정보도 갱신한다.
6. API는 성공 응답을 반환한다.

#### 시나리오 2. Atomic 정책의 배치 수집

1. 센서 게이트웨이가 여러 레코드가 담긴 배열을 전송한다.
2. 클라이언트는 정책을 생략하거나 `atomic`을 선택한다.
3. 백엔드는 전체 레코드를 모두 검증한 뒤 저장 여부를 결정한다.
4. 한 레코드라도 잘못되면 전체 배치를 실패로 처리한다.
5. API는 구조화된 오류를 반환하고 아무 데이터도 저장하지 않는다.

#### 시나리오 3. Partial 정책의 배치 수집

1. 센서 게이트웨이가 여러 레코드가 담긴 배열을 전송한다.
2. 클라이언트는 `partial`을 선택한다.
3. 백엔드는 레코드를 개별적으로 검증한다.
4. 유효한 레코드는 저장한다.
5. 유효하지 않은 레코드는 index 기반 오류와 함께 반환한다.
6. 응답에는 성공 건수와 실패 건수가 함께 포함된다.

#### 시나리오 4. 운영자의 이력 조회

1. 운영자가 특정 시리얼 번호와 시간 범위를 선택한다.
2. 프론트엔드가 조회 API를 호출한다.
3. 백엔드는 시리얼 번호, 모드, 시간 조건으로 데이터를 필터링한다.
4. 백엔드는 최신순의 paginated 응답을 반환한다.

#### 시나리오 5. 고장 센서 탐지

1. 상태 평가 프로세스가 센서별 최신 수신 시각을 확인한다.
2. 현재 서버 시각과 마지막 모드 기준 기대 주기를 비교한다.
3. 임계 시간을 넘기면 해당 센서를 이상 상태로 표시한다.
4. 조회 API 또는 상태 API를 통해 최신 상태가 노출된다.

#### 시나리오 6. 운영자의 모드 변경 요청

1. 운영자가 특정 센서의 모드 변경을 요청한다.
2. 백엔드는 모드 변경 요청 이력을 저장한다.
3. 이후 센서가 전송한 텔레메트리에서 실제 적용된 모드가 보고된다.
4. 백엔드는 요청된 모드와 실제 보고된 모드를 비교할 수 있다.

### 백엔드 지원 관점의 UI/UX 고려사항

이 문서는 백엔드 중심 문서이지만, API는 프론트엔드가 일관된 사용자 경험을 구현할 수 있도록 도와야 한다.

- 시간 값은 브라우저가 안정적으로 로컬라이즈할 수 있도록 일관된 직렬화 형식으로 제공되어야 한다.
- 조회 응답은 pagination과 응답 구조가 예측 가능해야 한다.
- 센서 상태는 프론트엔드가 timestamp를 직접 해석하지 않아도 되도록 명시적으로 제공되어야 한다.
- 오류 응답은 화면 표시와 디버깅 모두에 활용될 수 있도록 충분히 설명적이어야 한다.

### 센서 데이터 계약

#### 입력 형태

수집 엔드포인트는 다음 두 가지 최상위 payload 형태를 지원한다.

단일 객체:

```json
{
  "serial_number": "SENSOR-A-1004",
  "timestamp": "2024-05-23T08:30:00Z",
  "mode": "NORMAL",
  "temperature": 24.5,
  "humidity": 50.2,
  "pressure": 1013.2,
  "location": {
    "lat": 37.5665,
    "lng": 126.9780
  },
  "air_quality": 42
}
```

배열:

```json
[
  {
    "serial_number": "SENSOR-B-2024",
    "timestamp": "2024-05-23T17:40:00+09:00",
    "mode": "EMERGENCY",
    "temperature": 28.1,
    "humidity": 65.4,
    "pressure": 1009.5,
    "location": {
      "lat": 35.1796,
      "lng": 129.0756
    },
    "air_quality": 88
  }
]
```

#### 필드 의미

- `serial_number`: 센서 도메인에서의 고유 식별자
- `timestamp`: 서버 수신 시각이 아니라 센서가 데이터를 생성한 시각
- `mode`: 측정 시점의 센서 작동 모드
- `location`: 해당 측정값에 연결된 위치 정보
- 기타 metric 필드: 표시 및 모니터링에 사용되는 환경 측정값

### 시간 정규화 정책

시간 처리 전략은 이 과제에서 가장 중요한 설계 결정 중 하나다.

#### 내부 기준 규칙

- 들어오는 센서 timestamp는 timezone-aware 하게 파싱되어야 한다.
- 파싱된 시각은 내부의 단일 canonical 표현으로 변환되어야 한다.
- 모든 비교, 필터링, 정렬, 상태 판별 로직은 이 canonical 시각을 사용해야 한다.

#### 저장되는 시간 필드

- `sensor_timestamp`: 센서가 생성한 이벤트 시각의 정규화 결과
- `server_received_at`: 백엔드가 해당 레코드를 수신한 시각
- `raw_timestamp`: 클라이언트가 보낸 원본 문자열

#### 센서 시각과 서버 시각을 모두 보존해야 하는 이유

- 센서 시각은 실제 비즈니스 진실값과 이력 순서를 표현한다.
- 서버 시각은 수집 지연과 운영 시점 기준 판단에 필요하다.
- raw 문자열은 mixed timezone 입력의 문제를 진단할 때 유용하다.

### API 명세

#### 1. 수집 API

- Method: `POST`
- Path: `/api/v1/readings`

Request body:

- 센서 측정 객체 하나
- 또는 센서 측정 객체 배열

```json
  "rejected_count": 0,
  "errors": []
}
```


- 배치 정책은 request body가 파싱 가능한 경우에만 적용된다.

### 센서 상태 판별 정책

이 과제의 상태 판별은 고급 이상 탐지가 아니라 데이터 누락 기반 건강 상태 평가에 초점을 둔다.

#### 핵심 규칙

- 마지막으로 확인된 모드가 `NORMAL`이면 다음 데이터는 10분 이내에 도착해야 한다.
- 마지막으로 확인된 모드가 `EMERGENCY`이면 다음 데이터는 10초 이내에 도착해야 한다.
- 설정된 임계 시간을 초과하도록 새 데이터가 도착하지 않으면 해당 센서는 이상 상태로 판단한다.

#### 권장 임계값

- `NORMAL`: 12분 동안 데이터가 없으면 이상
- `EMERGENCY`: 30초 동안 데이터가 없으면 이상

#### 평가 전략 옵션

이 프로젝트는 다음 방식 중 하나로 상태 평가를 구현할 수 있다.

- 새 데이터가 들어올 때 최신 상태를 갱신하고, 필요 시 주기적 재평가를 보조적으로 수행하는 방식
- 경량 스케줄러가 전체 센서를 주기적으로 점검하는 방식

과제는 메시지 큐를 제한하지만 scheduled work 자체를 금지하지 않으므로, 가벼운 스케줄링 기반 구현은 충분히 허용 가능한 선택이다.

### 오류 처리 및 검증 정책

#### 요청 수준 오류

예시:

- malformed JSON
- 지원하지 않는 최상위 payload 타입
- 지원하지 않는 ingestion mode 값

이런 오류는 전체 요청 실패를 유발한다.

#### 레코드 수준 오류

예시:

- 필수 필드 누락
- 지원하지 않는 enum 값
- 파싱 불가능한 timestamp
- 잘못된 `location` 구조

이런 오류는 다음과 같이 처리된다.

- `atomic` 모드에서는 전체 배치를 실패시킨다.
- `partial` 모드에서는 해당 레코드만 거절한다.

#### 오류 응답 원칙

- 배치 요청인 경우 실패한 레코드의 index를 식별해야 한다.
- 가능한 경우 문제 필드를 함께 식별해야 한다.
- 필요하다면 구현 단계에서 재시도 가능 여부 같은 메타데이터를 확장할 수 있다.

### 가정사항

- 센서는 모든 `timestamp` 문자열에 timezone 정보를 포함해서 보낸다.
- `serial_number`는 하나의 논리적 센서를 안정적으로 식별한다.
- 백엔드는 상태 판별을 위해 주기적 점검 작업을 수행할 수 있다.
- 외부 모드 제어는 실제 디바이스 전달이 완전 구현되지 않더라도 서버 측 요청 기록으로 표현할 수 있다.
- 프로젝트는 시간 및 시리얼 번호 기준 필터링이 가능한 영속 저장소를 사용한다.
- 인증 및 권한 관리는 구현 범위에 따라 단순화될 수 있다.

## Reflection

이 명세는 과제에서 특히 중요하게 평가될 가능성이 높은 시간 정규화와 운영 규칙의 명확성에 초점을 맞춘다. 가장 중요한 설계 결정은 센서 생성 시각과 서버 수신 시각을 분리해서 다루는 것이다. 이 구분이 없으면 조회 의미가 모호해지고, 네트워크 지연 때문에 상태 판별 결과가 왜곡될 수 있다.

배치 수집 정책을 고정값이 아니라 선택 가능한 정책으로 정의한 것도 의도적인 결정이다. 기본값을 `atomic`으로 두면 초기 구현이 단순해지고 테스트가 쉬워지며, 과제 범위 안에서 실수 가능성을 줄일 수 있다. 반면 `partial`을 옵션으로 열어두면 실제 IoT 운영 환경에서 더 현실적인 선택지도 함께 제시할 수 있다. 즉, 구현 복잡도와 운영 유연성 사이의 균형을 문서 수준에서 먼저 확보한 셈이다.

센서 상태 판별 기준 역시 요구사항을 그대로 반복하지 않고 해석을 추가했다. 원문은 기대 주기만 제시하고 실제로 언제부터 고장으로 볼지 grace threshold를 정의하지 않는다. 하지만 현실 시스템에서 주기와 정확히 일치하지 않는다는 이유만으로 곧바로 이상으로 판단하면 오탐지가 많아질 수 있다. 따라서 이 명세는 작은 완충 구간을 둔 권장 기준을 함께 제안한다.

또한 이 설계는 과제의 기술 제약을 존중하여 인프라가 무거운 구성요소를 피한다. 메시지 큐 없이도 수집, 저장, 상태 평가, 조회를 충분히 설계할 수 있으며, 오히려 그 제한 덕분에 시스템의 구조와 책임이 더 분명해진다.

마지막으로 이 문서는 기술 중립적으로 작성되었지만, 실제 구현으로 바로 이어질 수 있을 정도로 구체적인 계약과 구조를 포함한다. 따라서 다음 단계인 `Pseudocode`에서는 이를 함수와 처리 흐름으로 쪼갤 수 있고, 이후 `Architecture` 단계에서는 동일한 규칙을 유지한 채 구체적인 스택으로 옮겨갈 수 있다.
